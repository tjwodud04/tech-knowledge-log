Approximate Nearest Neighbors (ANN) & Vector Search

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-06-29

내용

벡터 검색(ANN)은 고차원 임베딩에서 유사한 항목을 빠르게 찾는 기술로, 추천·검색·유사도 검색에서 실시간 응답과 비용 절감을 위해 필수적입니다.

핵심 아이디어는 정확한 선형 탐색(Brute-force)을 피하고 근사 구조를 사용해 자주 허용 가능한 오차 범위 내에서 최근접 이웃을 반환하는 것입니다. 주요 기법은 그래프 기반(HNSW), 공간 분할과 IVF(중심군 기반), 그리고 양자화(PQ, OPQ)입니다. HNSW는 노드 간 소규모 그래프를 탐색해 낮은 지연시간과 높은 정밀도를 제공하고, IVF+PQ(FAISS)는 클러스터에 나눈 뒤 각 클러스터를 저비용 코드로 압축해 대용량에 적합합니다.

실제 오픈소스/연구 예시는 Facebook의 FAISS(IVF/PQ/OPQ), Spotify의 Annoy(랜덤 투영 트리), Malkov의 HNSW(HNSWlib), Google의 ScaNN가 있습니다. 산업에서는 Pinecone/Milvus 같은 벡터 DB가 인프라를 단순화하고, Elastic/KNN 플러그인은 기존 검색 스택에 통합됩니다. 트레이드오프는 구축 시간·메모리·정확도·업데이트 용이성 간의 균형입니다.

활용

FAISS (IVF+PQ) 예제 — 대용량 인덱스에서 자주 사용됨 (GitHub: facebookresearch/faiss):

import numpy as np
import faiss
d=128
xb = np.random.rand(100000, d).astype('float32')
index = faiss.index_factory(d, "IVF65536,PQ32")
index.train(xb)
index.add(xb)
D,I = index.search(xb[:5], 5)

HNSWlib (저지연 추천) 예제:
import hnswlib
p = hnswlib.Index(space='cosine', dim=128)
p.init_index(max_elements=200000, ef_construction=200, M=16)
p.add_items(data)
labels, dist = p.knn_query(q, k=10)

Elastic 벡터 매핑 예: "mappings":{"properties":{"vec":{"type":"dense_vector","dims":768}}}

실무 패턴: 메타데이터 조인(필터 후 벡터 검색), 인크리멘털 리빌드, 샤딩·레플리카로 질의 지연 분산, recall/latency SLO 계측. 출처: FAISS GitHub, HNSW paper (Malkov+Yashunin), Annoy.

Content

Vector search (ANN) finds nearest embeddings fast by trading exactness for latency and cost — critical for recommendation, semantic search, and retrieval at scale.

The core idea is to avoid brute-force O(N) comparisons using approximate structures that return near-neighbors with high probability. Major algorithm families: graph-based (HNSW), partition-based (IVF), and quantization (PQ/OPQ). HNSW builds a navigable small-world graph for very low query latency and high recall; IVF clusters vectors and searches a subset of clusters; PQ compresses vectors into compact codes to reduce memory and I/O.

Popular implementations: FAISS (Facebook) supports IVF+PQ/OPQ and GPU acceleration, Annoy (Spotify) uses randomized trees for read-heavy workloads, HNSWlib implements Malkov’s HNSW for fast dynamic indexes, and ScaNN (Google) targets TPU/CPU optimizations. Vector DBs like Milvus or managed services (Pinecone) simplify production concerns like sharding, replication, and hybrid filtering.

Applications

FAISS (IVF+PQ) production snippet — common pattern for large offline indexes (source: facebookresearch/faiss):

import numpy as np
import faiss
d=128
xb = np.random.rand(100000, d).astype('float32')
index = faiss.index_factory(d, "IVF65536,PQ32")
index.train(xb)
index.add(xb)
D,I = index.search(xb[:5], 5)

HNSWlib — low-latency online use:
import hnswlib
p = hnswlib.Index(space='cosine', dim=128)
p.init_index(max_elements=200000, ef_construction=200, M=16)
p.add_items(data)
labels, dist = p.knn_query(q, k=10)

Elasticsearch mapping for vector fields:
{"mappings": {"properties": {"vec": {"type":"dense_vector","dims":768}}}}

Operational best practices: filter-by-metadata before vector search, periodic re-training/reindexing for drift, tune ef/search_k for recall vs latency, warm caches and measure recall/latency SLOs. Sources: FAISS repo, HNSW paper (Malkov & Yashunin), Annoy, ScaNN, Milvus.
