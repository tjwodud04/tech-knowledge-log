Breadth-First Search (BFS)

난이도 / Difficulty: Intermediate
날짜 / Date: 2025-11-24

내용

BFS는 그래프를 레벨 단위로 탐색해 최단 경로(간선 수 기준)나 연결성 확인에 적합한 기본 알고리즘으로, 대규모 시스템에서는 레벨 동기화(frontier), 양방향 탐색, 병렬화가 핵심입니다.

작동 원리: 시작 정점에서 큐(혹은 frontier)를 사용해 인접 정점을 방문하고 방문 플래그를 세워 중복을 막으며 레벨 단위로 확장합니다. 시간복잡도는 O(V+E), 메모리는 방문 비트맵 + frontier 크기입니다. 실무에선 단일-스레드 구현 대신 레벨 동기 구조(level-synchronous BFS)를 사용해 SIMD/GPU(예: Gunrock)나 분산 프레임워크(Pregel/Giraph, GraphX)에서 frontier를 병렬로 처리합니다.

엔지니어링 포인트: 대규모 소셜 그래프(수십억 엣지)에서는 양방향 BFS로 탐색 폭을 반감시키고, frontier 압축(bitmaps/CSR slices)과 캐시 친화적 메모리 레이아웃으로 메모리 대역폭을 최적화합니다. 또한 목적지 발견 시 즉시 탈출해 불필요한 레벨 확장을 줄이는 것이 실무적입니다.

활용

사례: Neo4j(그래프 DB)와 NetworkX, GPU/분산 연구 구현에서 사용
코드:
// Neo4j (4.x): unweighted shortest path uses BFS internally
MATCH (a:User {id:1}), (b:User {id:42})
MATCH p=shortestPath((a)-[*]-(b))
RETURN p

# NetworkX (v2.8+): BFS tree / edges
import networkx as nx
G=nx.read_edgelist('edges.txt', nodetype=int)
for u,v in nx.bfs_edges(G, source=0):
    process(u,v)

실무 패턴:
- 대규모 그래프: Gunrock (GPU) bfs 실행: ./bin/bfs market graph.mtx --src=0 --undirected (Gunrock, UC Davis 논문, v1.0+)
- 분산: Pregel/Pregel-유사 Apache Giraph(1.3)나 Spark GraphX(3.0)에서 frontier 메시지 전달 방식으로 확장
- Best practice: 양방향 BFS, depth limits(웹 크롤러), 방문 비트맵 압축으로 메모리/대역폭 최적화
참고: Gunrock(2015), Pregel 논문(2009), Neo4j GDS/shortestPath 기능 사용

Content

BFS explores a graph level-by-level, ideal for unweighted shortest paths and connectivity checks; in production, scaling requires frontier management, bidirectional search, and parallel implementations.

How it works: BFS starts from a source, uses a queue or frontier bitmap to enumerate neighbors level-by-level, marking visited nodes to avoid repeats. Complexity is O(V+E) and memory is dominated by visited state plus current frontier. In practice, single-thread BFS is replaced by level-synchronous parallel BFS for GPUs (e.g., Gunrock) or distributed systems (Pregel/Giraph, GraphX) where the frontier is processed in parallel.

Engineering concerns: For billion-edge social graphs, bidirectional BFS halves explored levels for point-to-point queries; frontier compression (bitmaps/CSR slices) and cache-friendly layouts reduce memory bandwidth. Early exit on target discovery and limiting depth for crawlers are common production optimizations.

Applications

Case: Used in Neo4j, NetworkX, Gunrock (GPU), Pregel/Giraph
Code:
// Neo4j (4.x): built-in shortestPath uses BFS semantics for unweighted paths
MATCH (a:User {id:1}), (b:User {id:42})
MATCH p=shortestPath((a)-[*]-(b))
RETURN p

# NetworkX (v2.8+): bfs_edges
import networkx as nx
G = nx.read_edgelist('edges.txt', nodetype=int)
for u,v in nx.bfs_edges(G, source=0):
    handle_edge(u,v)

Production patterns:
- GPU: Gunrock (./bin/bfs market graph.mtx --src=0 --undirected) used in research (Gunrock, UC Davis, 2015)
- Distributed: Pregel-style message-passing in Apache Giraph (1.3) or Spark GraphX (3.0) for large graphs
- Best practices: use bidirectional BFS for point queries, compress visited bitmaps, prefer frontier-level parallelism and early exit to minimize work
