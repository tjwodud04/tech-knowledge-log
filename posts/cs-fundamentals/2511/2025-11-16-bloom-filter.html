Bloom Filter (멤버십 검사용 확률적 자료구조)

난이도 / Difficulty: Intermediate
날짜 / Date: 2025-11-16

내용

Bloom Filter는 메모리/디스크를 적게 쓰면서 '멤버십(집합 포함 여부)'을 빠르게 검사하는 확률적 자료구조입니다. 대규모 분산 시스템에서 캐시 미스, 중복 제거, 네트워크 필터링 등에 자주 쓰입니다.

작동 원리는 고정된 비트 배열(bit array)과 k개의 해시 함수를 사용해 항목을 삽입할 때 여러 비트를 1로 설정하고, 조회시 모든 해당 비트가 1이면 ‘있을 가능성(참)’을, 하나라도 0이면 ‘없음(거짓)’을 반환합니다. 따라서 false positive(거짓 긍정)는 있으나 false negative(거짓 부정)는 없습니다. 필터 크기 m, 예상 원소 수 n, 허용 오탐률 p에 대해 m = - (n * ln p) / (ln 2)^2, k = (m / n) * ln 2로 계산해 용량을 튜닝합니다. 삭제가 필요하면 Counting Bloom Filter(카운터 배열)나 Cuckoo Filter 같은 변형을 사용합니다.

실무적으로는 디스크 I/O와 네트워크 비용 절감 목적이 큽니다. 예컨대 HBase는 SSTable 블록을 읽기 전에 Bloom Filter로 필터링해 불필요한 디스크 읽기를 줄입니다(특히 대용량 키-값 저장소에서 효과적). RedisBloom 모듈은 Redis에 Bloom 연산을 추가해 실시간 애플리케이션에서 사용됩니다.

활용

사례: Google/Chrome의 Safe Browsing(악성 URL 필터링, 논문/구현 참조), Apache HBase(블록 단위 Bloom Filter 적용), Apache Cassandra(SSTable 레벨 Bloom Filter), RedisBloom(모듈: RedisBloom v2.x)
코드:
# 단순 Bloom Filter 예 (bitarray, mmh3)
from bitarray import bitarray
import mmh3
m, k = 10000, 4
bits = bitarray(m); bits.setall(0)
def add(item):
  for i in range(k): bits[mmh3.hash(item, i) % m] = 1
def exists(item):
  return all(bits[mmh3.hash(item, i) % m] for i in range(k))

RedisBloom 명령:
BF.RESERVE mybf 0.01 100000
BF.ADD mybf "user:123"
BF.EXISTS mybf "user:123"

실무 패턴:
- 예상 n과 허용 p를 기반으로 m,k 선계산(서비스 재시작 시 유지)
- 삭제 필요 시 Counting Bloom 또는 Cuckoo Filter 사용
- 보안·정확성 요구가 높은 경우 Bloom 결과로 2차 검증(예: 실제 DB 조회)
참조: HBase 공식문서, RedisBloom 모듈 문서, Google Safe Browsing 논문/구현

Content

A Bloom Filter is a space-efficient probabilistic data structure for membership tests. It's widely used in large production systems to reduce I/O, network calls, and to quickly filter candidates.

It uses a fixed-size bit array and k hash functions: inserting sets k bits to 1, querying checks those bits; all 1s means 'possibly present', any 0 means 'definitely absent'. Bloom Filters allow false positives but not false negatives. Given expected items n and target false-positive probability p, size m and hash count k are chosen by m = -n ln p / (ln 2)^2 and k = (m/n) ln 2. For deletions or counting, use Counting Bloom Filters or alternatives like Cuckoo Filters.

In production the major benefit is saving expensive operations—disk reads, network lookups, or expensive computations. HBase uses Bloom filters per HFile/SSTable to avoid unnecessary block reads. RedisBloom provides BF.ADD/BF.EXISTS commands for real-time services. Guava (com.google.common.hash.BloomFilter) offers a robust Java implementation used in many JVM services.

Applications

Case: Used in Google Chrome Safe Browsing (malicious URL filtering), Apache HBase (HFile block Bloom filters), Apache Cassandra (SSTable-level Bloom filters), RedisBloom module (RedisBloom v2.x)
Code:
# minimal example (bitarray + mmh3)
from bitarray import bitarray
import mmh3
m, k = 10000, 4
bits = bitarray(m); bits.setall(0)
def add(item):
  for i in range(k): bits[mmh3.hash(item, i) % m] = 1
def exists(item):
  return all(bits[mmh3.hash(item, i) % m] for i in range(k))

Redis commands:
BF.RESERVE mybf 0.01 100000
BF.ADD mybf "user:123"
BF.EXISTS mybf "user:123"

Production patterns:
- Precompute m,k from expected n and p; persist filter state if needed
- Combine Bloom checks with a secondary definitive lookup for critical paths
- Use Counting or Cuckoo filters when deletions or lower FPR are required
