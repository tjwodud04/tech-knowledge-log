Raft Consensus Algorithm

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-07-03

내용

Raft는 분산 시스템에서 복제 로그를 통해 일관성을 보장하는 합의 알고리즘으로, 이해하기 쉽고 실무에 널리 채택되어 장애 복구와 구성 변경에 강합니다.

Raft의 핵심은 세 가지: 리더 선출(leader election), 로그 복제(log replication), 안전성(safety)입니다. 시스템은 기간(term)으로 구분되며, 각 기간에 하나의 리더가 존재해 클라이언트 요청을 받아 로그 항목을 다른 팔로워에게 appendEntriesRPC로 전파합니다. 리더는 과반수 승인을 받은 항목만 커밋하고 순서를 보장합니다. 안전성은 과거에 커밋된 로그가 다른 리더에 의해 덮어쓰이지 않도록 term과 인덱스 조건을 통해 지켜집니다.

운영 관점에서는 선거 타임아웃(election timeout)과 하트비트(heartbeat) 주기 설정, 스냅샷(snapshot)과 로그 압축(compaction), 멤버십 변화(joint consensus)가 중요합니다. 스냅샷은 로그가 비대해질 때 상태 머신의 스냅샷을 저장하고 오래된 로그를 제거해 복구 시간과 디스크 사용량을 줄입니다. 멤버십 변경은 안전하게 노드를 추가/제거하기 위해 두 단계의 조인트 컨센서스(joint consensus)를 사용합니다.

활용

실제 사용 사례:
- etcd (CoreOS): Kubernetes의 키-값 저장소로 etcd 서버는 Go의 etcd/raft 라이브러리를 사용합니다. 프로덕션에서는 토폴로지와 electionTimeout, snapshotCount를 튜닝합니다.
예: etcd 시작 플래그
"etcd --name infra1 --initial-advertise-peer-urls http://10.0.0.1:2380 --listen-peer-urls http://0.0.0.0:2380 --listen-client-urls http://0.0.0.0:2379 --initial-cluster infra1=http://10.0.0.1:2380,infra2=http://10.0.0.2:2380"

- 라이브러리 사용(Go 등원시): etcd/raft의 낮은 수준 API로 제안하는 예:
"raftNode.Propose(context.TODO(), []byte(\"set x=1\"))"

- HashiCorp Consul, CockroachDB도 Raft 변형을 사용합니다(참고: Ongaro & Ousterhout, 'Search for an Understandable Consensus Algorithm' 2014). 실무 패턴: 과반수 노드 유지, 스냅샷 주기 설정, 네트워크 분할 대비 read-only 모드/leader-lease 활용.

Content

Raft is a replication-based consensus algorithm that ensures consistency across distributed state machines; it's designed to be easier to understand and implement, and is widely used in production systems.

The core of Raft: leader election, log replication, and safety. Time is divided into terms; in each term one leader handles client requests and replicates log entries to followers using AppendEntries RPCs. A leader considers an entry committed once a majority acknowledges it; committed entries are applied to the state machine in order. Safety is enforced by leader election rules and term/index checks so that a new leader never loses previously committed entries.

Operational concerns include election timeout and heartbeat tuning, log compaction via snapshots, and safe membership changes. Snapshots reduce recovery time and disk usage by serializing state and discarding old logs. Membership changes use joint consensus (two-phase) to avoid losing majority quorums during reconfiguration.

Applications

Real-world use cases:
- etcd (CoreOS): used by Kubernetes for cluster metadata. etcd server uses the Go etcd/raft implementation; operators tune electionTimeout and snapshotCount for stability.
Example etcd startup flags:
"etcd --name infra1 --initial-advertise-peer-urls http://10.0.0.1:2380 --listen-peer-urls http://0.0.0.0:2380 --listen-client-urls http://0.0.0.0:2379 --initial-cluster infra1=http://10.0.0.1:2380,infra2=http://10.0.0.2:2380"

- Low-level API (Go): proposing entries with etcd/raft:
"node.Propose(ctx, []byte(\"put key val\"))"

- Other adopters: HashiCorp Consul (service discovery), CockroachDB (distributed SQL) use Raft variants. Best practices: maintain odd-numbered clusters or strict majority, enable snapshots and log compaction, monitor election frequency, and use leader leases for low-latency reads. References: Ongaro & Ousterhout, 2014 (Raft paper); etcd repo (github.com/etcd-io/etcd).
