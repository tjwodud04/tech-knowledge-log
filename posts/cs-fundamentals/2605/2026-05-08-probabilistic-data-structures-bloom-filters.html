Bloom Filters (확률적 자료구조: 블룸 필터)

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-05-08

내용

블룸 필터는 적은 메모리로 집합 멤버십을 빠르게 테스트하는 확률적 자료구조로, 캐시/디스크 조회 감소와 분산 시스템의 네트워크 비용 절감에 실무적으로 유용합니다.

비트 배열 m과 k개의 해시 함수를 사용해 항목 n을 삽입하면 해당 비트들을 1로 설정합니다. 멤버십 질의는 모든 k 위치가 1인지 검사해 '있을 수 있음' 또는 '확실히 없음'을 반환하며, 거짓긍정(false positive)은 허용되지만 거짓부정(false negative)은 없습니다. 확률 p는 약 p ≈ (1 - e^{-k n / m})^k 이고 최적 k = (m/n) ln 2입니다. 실무에서는 메모리/정확도 트레이드오프를 계산해 m과 k를 결정합니다.

확장성 문제는 Counting Bloom Filter(삭제지원), Scalable/Partitioned Bloom Filter(동적 확장), Cuckoo Filter(삭제 및 더 낮은 FP) 등으로 해결합니다. 또한 Bloom 필터는 비트단위 OR/AND로 합치거나 분산 캐시에서 로컬 체크 후 원격 조회를 줄이는 데 자주 쓰입니다. 주요 사례로는 HBase/Cassandra/Hadoop 계열의 SSTable 레벨 필터, Chrome Safe Browsing(프리픽스 집합), RedisBloom 모듈 등이 있습니다.

활용

실제 사용 사례:
- 사례: HBase에서 열패밀리 수준 Bloom 필터 설정 (HBase 2.4)
코드:
ColumnFamilyDescriptorBuilder cfd = ColumnFamilyDescriptorBuilder.newBuilder("cf".getBytes());
cfd.setBloomFilterType(BloomType.ROW);
admin.createTable(TableDescriptorBuilder.newBuilder(TableName.valueOf("tbl")).setColumnFamily(cfd.build()).build());

설명: HBase/Hadoop 계열은 SSTable 읽기 시 I/O를 줄이기 위해 블룸 필터를 사용합니다.

- 사례: RedisBloom 모듈 (RedisBloom v2.x)
코드:
# 모듈 설치 후
BF.RESERVE users 0.01 100000
BF.ADD users user:12345
BF.EXISTS users user:99999

설명: RedisLabs의 RedisBloom은 실무에서 빠른 존재 체크/캐시 프리필터로 사용됩니다.

- 사례: 라이브러리 (Guava 31.x)
코드:
BloomFilter<String> bf = BloomFilter.create(Funnels.stringFunnel(Charset.forName("UTF-8")), 100000, 0.01);
bf.put("id:1");
bf.mightContain("id:2");

실무 패턴: 예상 삽입량/목표 FP율로 생성, 주기적 리빌드(과도한 삽입 후), 필요시 Counting/Scalable 변종 사용. 참고: Bloom(1970), Broder & Mitzenmacher(2004) 서베이.

Content

A Bloom filter is a space-efficient probabilistic data structure for set membership tests; it reduces disk/DB lookups and network calls in production by quickly filtering negatives with low memory cost.

A Bloom filter uses an m-bit array and k independent hash functions. To insert n items, set k bit positions per item. A membership test checks if all k bits are 1: if not, the item is definitely absent; if yes, it is probably present (possible false positive, no false negatives). False positive probability p ≈ (1 - e^{-k n / m})^k; optimal k ≈ (m/n) ln 2. Operators like bitwise OR/AND enable merge across partitions.

Practical variants: Counting Bloom Filters (support deletions), Scalable/Partitioned Bloom Filters (grow without full rebuild), and Cuckoo Filters (support deletion and often lower FP for similar space). Bloom filters are widely used as cache pre-filters and SSTable-level filters in distributed storage to avoid expensive I/O or cross-node lookups.

Applications

Real-world use case:
- Case: HBase (and Bigtable-inspired stores) use Bloom filters per column family to reduce HDFS/SSTable reads.
Code (HBase 2.4):
ColumnFamilyDescriptorBuilder cfd = ColumnFamilyDescriptorBuilder.newBuilder("cf".getBytes());
cfd.setBloomFilterType(BloomType.ROW);
admin.createTable(TableDescriptorBuilder.newBuilder(TableName.valueOf("tbl")).setColumnFamily(cfd.build()).build());

Production note: used in HBase/Hadoop to avoid disk seeks when looking up rows.

- Case: RedisBloom module (RedisLabs, RedisBloom v2.x)
Code:
# create filter with 1% fp and capacity 100k
BF.RESERVE users 0.01 100000
BF.ADD users user:12345
BF.EXISTS users user:99999

Production patterns: use Bloom as a fast negative cache before hitting DB, monitor FP growth, rebuild or use Scalable/Counting variants when over capacity.

- Case: Guava BloomFilter (Guava 31.x) for in-process pre-checks
Code:
BloomFilter<String> bf = BloomFilter.create(Funnels.stringFunnel(Charset.forName("UTF-8")), 100000, 0.01);
bf.put("id:1");
bf.mightContain("id:2");

References: Bloom 1970; Broder & Mitzenmacher survey (2004); Chrome Safe Browsing and many distributed DBs adopt Bloom-like prefix/filters in production.
