Named Entity Recognition (NER)

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-05-18

내용

NER은 텍스트에서 사람·기관·장소·시간 등 고유명사를 자동으로 인식·분류하는 기술로, 검색·지식그래프·고객지원 등 실무 시스템에서 핵심 전처리 단계로 사용됩니다.

작동 원리: NER은 일반적으로 시퀀스 라벨링 문제로 다뤄집니다. 입력은 토큰화(단어·서브워드), 라벨 스킴(BIO/BILOU)로 변환된 뒤 모델이 각 토큰에 엔티티 태그를 예측합니다. 전통적 파이프라인은 품사·의존구문·사전(가제트리어)을 활용한 CRF/Feature-based 모델이었고(Lafferty 2001 계열), 연구는 Lample et al. (2016)의 BiLSTM-CRF, 그리고 Devlin et al. (2019) BERT 기반의 미세조정(fine-tuning)으로 발전했습니다.
운영적 고려사항: 프로덕션에서는 서브워드 분할(WordPiece)로 인한 라벨 정렬, 실시간 지연(latency), 엔티티 정규화(앵커링/KB 링크), 드리프트 감지(F1/precision 변화) 등이 중요합니다. CoNLL-2003 같은 표준 데이터로 초기 성능을 체크하되, 도메인 라벨링(parallel annotation)과 active learning을 통해 레이블 편향을 줄입니다.

활용

실제 사용 사례:
- 사례: Explosion의 spaCy (프로덕션 NLP 파이프라인)
코드:
# spaCy v3.5 + en_core_web_trf (transformer 기반)
  import spacy
  nlp = spacy.load('en_core_web_trf')
  doc = nlp('Apple acquired Beats in 2014 .')
  for ent in doc.ents:
      print(ent.text, ent.label_)

실무 패턴: 엔티티 룰(EntitiyRuler)로 고정 패턴 보강, nlp.pipe로 배치 처리, model.to_onnx/torchscript로 서빙.

- 사례: Hugging Face Transformers (token-classification) used at scale
코드:
# transformers v4.33
  from transformers import pipeline
  ner = pipeline('token-classification', model='dslim/bert-base-NER', grouped_entities=True)
  ner('Google opened a new office in Seoul .')

실무 패턴: subword-to-token 매핑, grouping 옵션으로 엔티티 합치기, GPU 배치 처리.

- 사례: AWS Comprehend / Amazon Comprehend Medical in production
명령/사용법:
aws comprehend detect-entities --text "Patient reports fever and cough" --language-code en

실무 패턴: 관리형 API로 빠른 통합, 개인정보 엔티티 마스킹, 모니터링을 통한 모델 교체 정책.

참고: Lample et al., 2016 (BiLSTM-CRF), Devlin et al., 2019 (BERT) 및 spaCy/HuggingFace 공식 문서.

Content

NER detects and classifies named entities (people, orgs, locations, dates) in text; it’s a foundational component for search, knowledge graphs, customer support automation, and analytics pipelines.

How it works: NER is framed as sequence labeling. Text is tokenized (word or subword), labels use schemes like BIO/BILOU, and a model predicts labels per token. Classic systems used feature-rich CRF models; modern pipelines use neural models—BiLSTM-CRF (Lample et al., 2016) and Transformer fine-tuning (BERT, Devlin et al., 2019). Key engineering issues include aligning subword predictions to original tokens, handling out-of-vocabulary entities, and doing entity linking/normalization to a KB.
Production details: pipelines optimize for latency (batching, ONNX/TorchScript), monitoring (per-entity F1 drift), and continuous improvement (active learning, human-in-the-loop). Training often starts with CoNLL-2003 or OntoNotes then adapts with domain-specific annotations.

Applications

Real-world use case:
- Case: spaCy in production NLP stacks (Explosion)
Code:
# spaCy v3.5 with transformer-backed model
  import spacy
  nlp = spacy.load('en_core_web_trf')
  doc = nlp('Amazon acquired Whole Foods in 2017.')
  print([(e.text,e.label_) for e in doc.ents])

Production patterns: use EntityRuler for deterministic matches, nlp.pipe for high-throughput, export to ONNX for low-latency serving.

- Case: Hugging Face Transformers token-classification (widely used in research & startups)
Code:
# transformers v4.33
  from transformers import pipeline
  ner = pipeline('token-classification', model='dslim/bert-base-NER', grouped_entities=True)
  ner('Microsoft announced new Surface devices in October .')

Production patterns: map subword labels back to tokens, use grouped_entities to merge spans, optimize with mixed-precision and batching.

- Case: Amazon Comprehend / Comprehend Medical in production
Command:
aws comprehend detect-entities --text "Patient has hypertension" --language-code en

Production patterns: managed services for quick integration, preprocess to mask PHI, and implement drift alerts and periodic re-annotation.

References: Lample et al. 2016 (BiLSTM-CRF), Devlin et al. 2019 (BERT), spaCy/HuggingFace docs.
