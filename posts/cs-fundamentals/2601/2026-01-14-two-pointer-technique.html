Two-Pointer Technique / 투 포인터 기법

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-01-14

내용

두 포인터 기법은 정렬된 시퀀스에서 두 개의 인덱스를 이동시키며 조건을 만족하는 구간이나 교집합/병합을 찾는 알고리즘 패턴으로, 대용량 검색·쿼리 실행·스트리밍 시스템에서 비용을 낮추고 I/O를 절감하기 위해 널리 쓰입니다.

작동 원리: 두 개(또는 다수)의 포인터(p, q)를 각각 시퀀스의 시작에 두고 비교 결과에 따라 한쪽 혹은 양쪽 포인터를 전진시킵니다. 예: 정렬된 두 리스트의 교집합은 양쪽 포인터가 가리키는 값이 같을 때 결과에 추가하고, 작으면 작은 쪽을 한 칸 전진시키는 방식으로 O(n+m) 시간에 처리됩니다. 변형으로는 고정 길이 창을 이동하는 슬라이딩 윈도우(연속 합/최대 길이 부분배열), 서로 다른 속도로 진행하며 충돌 검사하는 두 포인터(예: 롱런 압축 해제) 등이 있습니다.
실전 최적화: 입력이 크게 편향되어 있으면 galloping/exponential search(긴 리스트를 빠르게 건너뛰기) 또는 skip pointers/skip lists를 사용해 희소-조밀 불균형을 줄입니다. 밀도가 높을 때는 비트셋(bitset)으로 변환해 랭크/선택 연산을 사용하는 편이 더 빠릅니다.

활용

사례: Apache Lucene / Elasticsearch에서의 문서 ID 교집합
코드:
# 정렬된 두 posting list의 교집합 (실무 단순화 버전)
def intersect(a,b):
    i=j=0
    res=[]
    while i<len(a) and j<len(b):
        if a[i]==b[j]: res.append(a[i]); i+=1; j+=1
        elif a[i]<b[j]: i+=1
        else: j+=1
    return res

Where: ConjunctionDISI 및 PostingsEnum 상호작용(예: Lucene 8.x/9.x)에서 AND 쿼리의 핵심 연산으로 사용됩니다. 참고 소스: Apache Lucene releases (e.g., lucene 8.11.0 ConjunctionDISI).
사례2: PostgreSQL의 Merge Join (src/backend/executor/nodeMergejoin.c, PostgreSQL v13+)는 정렬된 두 소스에 대해 두 포인터를 이용해 I/O와 비교를 최소화합니다.
실무 패턴: 입력이 정렬되어 있음을 보장, 편향된 길이에는 galloping/binary-advance 사용, 밀집도 높으면 비트셋으로 전환, 대용량 외부 정렬/병합 시 메모리 버퍼 크기와 I/O 배치 고려.

Content

The two-pointer technique advances two indices over sequences (often sorted) to find matches, windows, or merges with linear-ish cost — crucial for search engines, DB join operators, and streaming analytics to reduce CPU and I/O.

How it works: Place pointers i and j at sequence starts and compare targets. For sorted-list intersection increment the smaller pointer; when equal emit a result. Sliding-window is a related pattern where left/right pointers expand/contract to maintain invariants (sum, unique count). Variants include multi-pointer merges (k-way merge), two-phase iteration (skip/advance APIs), and exponential/galloping search to jump in long lists.
Optimizations in production: use skip pointers or index-level skip lists (Lucene) to avoid linear scans across sparse lists; switch to bitsets/bitmap indices when posting density is high (faster rank/select operations); use branchless or SIMD-accelerated compares for CPU-heavy loops. For extremely skewed sizes, advance the larger list with binary/exponential search to reduce work.

Applications

Case: Lucene/Elasticsearch postings intersection
Code:
# simple intersect of sorted docIDs
def intersect(a,b):
    i=j=0
    out=[]
    while i<len(a) and j<len(b):
        if a[i]==b[j]: out.append(a[i]); i+=1; j+=1
        elif a[i]<b[j]: i+=1
        else: j+=1
    return out

Where: Lucene (ConjunctionDISI, releases 8.x/9.x) implements posting-list intersections with two-pointer base logic plus skip/galloping; Elasticsearch uses Lucene under the hood. Case2: PostgreSQL merge join (nodeMergejoin.c in PostgreSQL v12+) uses a two-pointer merge over sorted inputs for efficient joins. Production patterns: ensure sorted inputs or maintain order during ingestion, apply galloping/skip when lists are size-skewed, convert to bitset for dense posting lists, and tune memory buffers for external merges.
