Breadth-First Search (BFS)

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-01-26

내용

BFS(너비 우선 탐색)는 그래프/트리에서 시작 노드로부터 동일한 거리(레벨)를 먼저 방문하는 탐색 기법으로, 비가중치 최단경로, 레벨 집계, 연결성 검사 등 실무 시스템에서 빈번히 사용됩니다.

작동 원리: BFS는 FIFO 큐를 사용해 시작 노드에서부터 인접 노드를 차례로 확장하며 레벨 단위로 탐색합니다. 각 정점은 한 번만 방문(visited 마킹)하며 시간복잡도는 O(V+E), 공간복잡도는 최대 O(V)입니다. 변형으로 경로 복원(parent 포인터), 다원소 시작(multi-source BFS), 레벨별 집계, 그리고 제한 깊이(최단 경로 상한)가 있습니다.
실무적 고려사항: 대규모 그래프(수십억 엣지)에서는 메모리와 캐시 지역성, 분산/병렬화가 중요합니다. Graph500, Pregel(및 Apache Giraph), GraphX, Gunrock(GPU BFS) 같은 시스템은 BFS를 고성능 환경에서 구현한 사례입니다. 또한 Neo4j(Traversal API)와 NetworkX(NetworkX 2.x)는 엔지니어들이 프로덕션과 분석 파이프라인에서 빠르게 적용하는 라이브러리입니다.

활용

실제 사용 사례:
- 사례: Neo4j(그래프 DB)에서 친밀도/최단경로 탐색
코드:
// Neo4j 4.x
MATCH (a:User {id:0}), (b:User {id:42})
CALL apoc.algo.shortestPath(a,b) YIELD path
RETURN path

- 사례: NetworkX를 이용한 분석 파이프라인(NetworkX 2.8)
코드:
import networkx as nx
G = nx.read_edgelist('edges.txt')
pred = nx.single_source_shortest_path_length(G, 0)

실무 패턴:
- Multi-source BFS: 여러 시작점에서 동시 탐색해 거리 계산 비용 절감(예: social graph distance)
- 분산 BFS: Pregel/Giraph 스타일의 superstep로 엣지를 분할하여 메모리 스케일아웃
- GPU 가속: Gunrock (NVIDIA)으로 대규모 실시간 서치 성능 개선
참고: Graph500 벤치마크는 BFS 성능을 대규모 그래프에서 비교하는 표준입니다.

Content

Breadth-First Search (BFS) explores a graph level-by-level from a source node; it's the go-to for unweighted shortest paths, connectivity, and level-order analytics in production systems.

How it works: BFS uses a FIFO queue and a visited set to expand nodes by distance from the source. Each node is enqueued once; runtime is O(V+E) and memory O(V). Common variants include parent-pointer path reconstruction, multi-source BFS for simultaneous frontiers, depth-limited BFS, and bidirectional BFS to halve explored space for pairwise shortest paths.
Production considerations: At web scale, BFS must address memory, locality, and parallelism. Systems like Google's Pregel-inspired frameworks (Apache Giraph), Spark GraphX, and GPU library Gunrock provide scalable BFS implementations. Graph databases (Neo4j) expose traversal APIs for producer workflows; analytics stacks use NetworkX or SNAP for smaller graphs and distributed platforms for billion-edge graphs.

Applications

Real-world use case:
- Case: Neo4j (graph DB) for shortest-path and recommendations
Code:
// Neo4j 4.x example
MATCH (s:User {id:1}), (t:User {id:99})
CALL apoc.algo.shortestPath(s,t) YIELD path
RETURN path

- Case: NetworkX 2.8 for offline analytics
Code:
import networkx as nx
G = nx.read_edgelist('edges.txt')
dists = nx.single_source_shortest_path_length(G, source=0)

Production patterns:
- Multi-source BFS for fan-out-limited social graph queries (LinkedIn/Facebook-like workloads)
- Distributed BFS on Pregel/Giraph or GraphX for billion-edge graphs
- GPU-accelerated BFS (Gunrock, CUDA) for real-time path queries and benchmarks (Graph500)
References: Neo4j Traversal API, GraphX docs, Gunrock paper/Repo for GPU BFS optimizations.
