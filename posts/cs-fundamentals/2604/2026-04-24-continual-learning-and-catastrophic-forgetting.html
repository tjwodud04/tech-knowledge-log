Continual Learning & Catastrophic Forgetting

난이도 / Difficulty: Intermediate
날짜 / Date: 2026-04-24

내용

계속 학습(Continual Learning)은 모델이 새 데이터를 지속적으로 학습하면서 이전 지식을 잃지 않도록 하는 기술로, 프로덕션 시스템의 모델 업데이트·개인화·온디바이스 적응에서 필수적입니다.

작동 원리: 연속적 분포 변화(데이터 드리프트) 상황에서 표준 SGD로만 학습하면 새 작업에 적응하면서 이전 작업의 성능이 급격히 떨어지는 'catastrophic forgetting'이 발생합니다. 대표적 대응 방식은 (1) 정규화 기반 방법: Elastic Weight Consolidation(EWC, Kirkpatrick et al. 2017)처럼 이전에 중요하다고 판단된 파라미터를 고정(또는 페널티 적용)하여 급격한 변화 억제, (2) 리플레이 기반 방법: 샘플(혹은 생성 모델로 만든 샘플) 버퍼를 섞어 재학습하여 분포 보존, (3) 구조적 확장: Progressive Networks 등으로 네트워크 확장 후 연결을 통해 이전 지식 보존.
실무 관점: 구글의 온디바이스/연합학습(Gboard, Federated Learning)에서는 로컬 업데이트·주기적 서버 집계에서 잊힘 방지를 위해 정규화+리플레이 전략을 혼합합니다. 오픈소스 라이브러리(ContinualAI/avalanche)는 연구·프로덕션 프로토타입에 쓰기 좋습니다.

활용

실제 사용 사례:
- 사례: EWC (논문: Kirkpatrick et al., 2017)
코드:
# PyTorch: EWC 페널티 (간단화)
  loss = task_loss(output, target)
  ewc_loss = 0
  for name, param in model.named_parameters():
      ewc_loss += F[name] * (param - theta_star[name]).pow(2).sum()
  loss = loss + lambda_ * ewc_loss / 2

사용처: 연구·프로토타입에서 자주 재현 (PyTorch 1.8+ 호환)

- 사례: 리플레이 (iCaRL / 경험 리플레이)
코드:
# 간단한 온라인 리플레이 버퍼
  buffer.append((x,y))
  batch = random.sample(buffer, k=min(len(buffer), B))
  train_on(batch + new_batch)

사용처: 클래스-증분 학습(iCaRL, Rebuffi et al. 2017), Avalanche 라이브러리(ContinualAI: https://github.com/ContinualAI/avalanche)

- 사례: 온라인/스트리밍 모델 업데이트 (Vowpal Wabbit)
명령:
# vw로 스트리밍 업데이트: 모델을 계속 덮어쓰기
  vw -d stream.train -f model.vw --loss_function logistic --learning_rate 0.5

사용처: 생산 서비스에서 실시간 개인화·랭킹(아마존류), VW v8+에서 널리 사용

실무 패턴: 리소스 제약(온디바이스)일땐 정규화+경량 리플레이, 서버 측이면 대규모 샘플 리플레이/생성모델 복원 사용. Avalanche로 실험 후 경량화해 프로덕션 푸시.

Content

Continual learning enables models to learn from a stream of data without catastrophically forgetting previously learned tasks — crucial for production systems that require continuous adaptation and personalization.

How it works: When models train sequentially on non‑stationary data, naive SGD overwrites weights and causes catastrophic forgetting. Main technical strategies: (1) Regularization-based (e.g., Elastic Weight Consolidation, EWC) add quadratic penalties on changing important weights; (2) Replay-based stores or generates past examples and mixes them with new data during training; (3) Architectural approaches expand or isolate subnetworks (Progressive Networks) to preserve prior functions. Hybrid methods combine these (regularize important params + small replay buffer) to balance compute and memory.
In production: Google’s on-device/federated setups (e.g., Gboard adaptations using Federated Learning, McMahan et al. 2017) mitigate forgetting by combining local updates with server-side aggregation and constraints. Research-to-prod tooling includes Avalanche (ContinualAI) for experiments and Vowpal Wabbit for lightweight online updates in streaming systems.

Applications

Real-world use cases:
- Case: EWC (Kirkpatrick et al., 2017)
Code:
# PyTorch: simplified EWC penalty
  loss = task_loss(output, target)
  ewc = sum(F[name] * (param - theta_star[name]).pow(2).sum()
            for name,param in model.named_parameters())
  loss = loss + lambda_ * ewc / 2

Where used: academic reproductions and prototyping (PyTorch 1.8+)

- Case: Replay / iCaRL (Rebuffi et al., 2017)
Code:
buffer.append((x,y))
  batch = random.sample(buffer, k=min(len(buffer), B))
  train_on(batch + new_batch)

Where used: class-incremental learning experiments; Avalanche lib (pip install avalanche-lib, https://github.com/ContinualAI/avalanche)

- Case: Online updates in production (Vowpal Wabbit)
Command:
vw -d stream.train -f model.vw --loss_function logistic --learning_rate 0.5

Where used: low-latency personalization/ranking pipelines (VW v8+), supports streaming updates to reduce forgetting

Production patterns: use small replay + regularization on resource-limited devices; server-side replay or generative replay for large-scale services. Prototype in Avalanche, then optimize for latency/size before deployment.
